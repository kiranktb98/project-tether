⚓ Tether

Impact-analysis and edge-case engine for coding agents.
Stops Claude Code from breaking feature B while fixing feature A.

Python 3.11+ Claude Haiku Ollama support MIT License

Install

terminal
$ pip install tether-engine # Ollama backend (optional, for free local analysis) $ pip install tether-engine[ollama]

Requires Python 3.11+ and an Anthropic API key (for Haiku) — or a running Ollama server for free local mode.

Quickstart

1

Set your API key

$ echo "ANTHROPIC_API_KEY=sk-..." >> .env # or: export ANTHROPIC_API_KEY=sk-...
2

Initialize tether in your project

$ tether init Tether init Project name [my-project]: ratelimiter OK Config written to .tether/config.yaml OK Tether instructions written to CLAUDE.md Added .tether/ to .gitignore Tether is ready. Run tether bootstrap to build the feature ledger.
3

Bootstrap — scan codebase, propose features, generate tests

$ tether bootstrap tether bootstrap Step 1/4 — Scanning codebase... Found 3 Python files. Step 2/4 — Proposing features (Haiku)... Haiku proposed 3 features. Cost so far: $0.0037 Step 3/4 — User review... [editor opens with proposed feature list — edit, save, close] Step 4/4 — Generating edge cases and tests... Bootstrap complete. Features: 3 Edge cases: 24 (18 must_handle) Tests written: 18 Session cost: $0.0821 Ledger: .tether/ledger.yaml (v2)

Use --yes to auto-accept the proposed ledger without opening an editor.

4

Start the watcher in one terminal, Claude Code in another

terminal 1 — tether watch
$ tether watch Watching 3 features across 3 files. Budget: $5.00/session. Press Ctrl+C to stop.
terminal 2 — claude
$ claude > Claude Code reads CLAUDE.md automatically and follows tether workflow.

The Bell Curve

Every feature has an edge-case risk surface — the set of unusual inputs and conditions the code must handle. Tether maps this surface by assigning each edge case a frequency label:

Edge-case frequency distribution for a typical feature
HIGH >1%
common
4 cases must-handle
MEDIUM 0.01%
peak of the curve
6 cases must-handle
LOW <0.01%
rare
4 cases only if catastrophic
NEGLIGIBLE
tail
1 case skip
must-handle zone = HIGH + MEDIUM frequencies, plus any LOW case with a catastrophic rationale (data loss, security breach, system crash). These are the cases tether generates test stubs for.

The shape tells you about your feature's risk profile. A feature with many HIGH-frequency edge cases needs aggressive testing of common paths. A feature heavy in LOW/NEGLIGIBLE cases is well-understood and stable.

Confidence Intervals

When you run tether plan, Haiku assesses the risk level for every feature that touches the file you're about to change. The confidence tells you how much to trust that assessment.

Confidence is computed from the spread of individual feature risk levels:

🎯

High confidence

All features landed on the same (or adjacent) risk level. Haiku's assessment is reliable — act on it.

span ≤ 1 level → high

⚠️

Medium confidence

Risks span 2 levels (e.g. none → medium). Review PLAN.md carefully before proceeding.

span = 2 levels → medium

🔴

Low confidence

Risks span 3–4 levels (e.g. none → high). High variance — Haiku is uncertain. Read every risk note.

span ≥ 3 levels → low

📊

What to do with it

Low confidence doesn't mean the change is safe — it means tether is uncertain. Treat every medium/high risk as real.

What it looks like in tether plan

$ tether plan src/ratelimiter.py --intent "add Redis caching" Analyzing impact of change to src/ratelimiter.py... Plan written to .tether/PLAN.md Risk spread (3 features analyzed) none ██████████░░░░░░░░░░ 1 low ░░░░░░░░░░░░░░░░░░░░ 0 medium ████████████████████ 1 ← ask threshold high ████████████████████ 1 ← STOP Confidence: low high variance — Haiku is uncertain, review PLAN.md carefully Cost: $0.0094 STOP — plan contains medium/high risks. Read .tether/PLAN.md and ask the user before proceeding.

When you see STOP: Claude Code is instructed to halt and ask you before making any changes. Read .tether/PLAN.md — it contains the specific risks and mitigations Haiku identified.

The Feature Ledger

The feature ledger at .tether/ledger.yaml is tether's source of truth. Every feature lives here with its files, edge cases, test links, and history.

.tether/ledger.yaml (excerpt)
version: 12 features: - id: f1 name: Per-user Rate Limiting description: Sliding window rate limiter, 100 req/60s per user status: active files: - src/ratelimiter.py - src/middleware.py edge_cases: - id: e1 description: User hits the rate limit exactly at the boundary frequency: high must_handle: true test_file: .tether/tests/feature_f1/test_boundary.py - id: e2 description: Clock skew between server instances corrupts window frequency: low rationale: Rare but catastrophic — data loss if window resets incorrectly must_handle: true # low freq but catastrophic → escalated to must-handle

The must_handle flag is never set by the model — it's computed deterministically:

  • HIGH or MEDIUM frequency → always must_handle: true
  • LOW frequency + catastrophic rationale (data loss / security / crash) → must_handle: true
  • Everything else → must_handle: false

tether init

Interactive setup. Creates .tether/, writes .tether/config.yaml, and injects tether workflow rules into CLAUDE.md.

$ tether init

Run once per project. Safe to re-run — it updates the CLAUDE.md block in place.

tether bootstrap [--yes]

One-time codebase scan. Builds the initial feature ledger.

  1. Scans all Python files with tree-sitter (no LLM, instant)
  2. Sends the symbol map to Haiku to propose features
  3. Opens your $EDITOR to review and correct proposals
  4. Generates edge cases and pytest stubs for every must-handle case

Tip: Use --yes to skip the editor and auto-accept proposals. Good for CI or trying it quickly. Edit .tether/ledger.yaml afterward to correct any misidentified features.

Cost: ~$0.08 for a 3-file project, ~$1.50 for a 33-feature codebase. One-time only.

tether plan <file> --intent "..."

Run this before making a change. Predicts which features are at risk and at what severity.

$ tether plan src/ratelimiter.py --intent "replace in-memory window with Redis"

Writes .tether/PLAN.md. If any feature is at medium or high risk, Claude Code will STOP and ask you before proceeding. The plan includes:

  • Per-feature risk level and reason
  • Risk distribution chart + confidence interval
  • Specific mitigation suggestions for medium/high risks
  • Edge cases to handle if the change adds a new feature

tether coverage

Shows the edge-case bell curve for every feature and how much of the must-handle risk surface has test coverage.

$ tether coverage

Use this to spot features with gaps in their test coverage before making a change. The bar width for each frequency bucket shows the relative concentration of edge cases — the "shape" of that feature's risk profile.

Coverage % = must-handle edge cases with a test file / total must-handle edge cases.

tether watch

Long-running file watcher. On every file change:

  1. Computes a diff against the last known state
  2. Runs pytest for the affected features only (usually < 10s)
  3. Asks Haiku whether the diff aligns with the feature ledger (intent check)
  4. Checks for removed or changed function signatures (structural invariants)
  5. Writes a severity-graded note to .tether/DRIFT.md

Severity levels: none → soft → intentional → hard → critical

How drift works: When severity reaches hard or higher, tether writes a one-line pointer into CLAUDE.md. Claude Code reads CLAUDE.md at the start of every turn — so it sees the drift note immediately on its next message.

tether ask "<question>"

Natural-language Q&A against the feature ledger. No full-codebase search — just the structured metadata.

$ tether ask "which features would break if I change the RateLimiter constructor?" Consulting ledger (3 features)... ## Impact of changing RateLimiter constructor Changing the RateLimiter constructor signature would directly affect: - f1 Per-user Rate Limiting — the constructor sets limit and window_seconds; all callers (including middleware) must be updated. - f3 Request Middleware — instantiates _limiter = RateLimiter() at module level using default args. A signature change would break this import. f2 (JWT Auth) is not affected — it does not import from ratelimiter.py. confidence: high · features: f1, f3 · cost: $0.0048

Cost: ~$0.005 per question.

tether verify

Post-change validation. Run this after finishing a unit of work.

  • Re-runs all tests for features whose files changed this session
  • Auto-promotes building features to active when tests pass
  • Clears .tether/DRIFT.md if all tests pass
  • Snapshots the ledger and writes a session report to .tether/reports/

Config reference

Stored at .tether/config.yaml. All fields are optional — shown with defaults.

.tether/config.yaml
project: name: my-project root: . # where to scan watcher: provider: anthropic # "anthropic" | "ollama" model: claude-haiku-4-5-20251001 budget_usd_per_session: 5.0 ollama_base_url: http://localhost:11434 scan: exclude_dirs: [] # extra dirs to skip during bootstrap plan: edge_case_count: 8 # max edge cases per feature ask_threshold: medium # none|low|medium|high ledger: path: .tether/ledger.yaml tests_dir: .tether/tests/

Ollama (free, local)

Run tether entirely offline with a local Ollama server. No API key needed.

# Install Ollama, then pull a model $ ollama pull qwen2.5:7b $ ollama serve

Then set your config:

.tether/config.yaml
watcher: provider: ollama model: qwen2.5:7b

Recommended models (best to acceptable): qwen2.5:7b → mistral:7b → llama3.2:3b

Quality note: Ollama models produce slightly lower-quality edge cases and impact analysis than Haiku. For the bootstrap (one-time run), Haiku is recommended. Ollama works well for the continuous tether watch loop where many small checks happen throughout a session.
