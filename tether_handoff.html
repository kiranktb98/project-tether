1. What you are building

Tether is an impact-analysis and edge-case engine that sits alongside Claude Code (or any coding agent) and helps it make safer, more deliberate changes.

The pain it solves: when you ask a coding agent to add or modify a feature, it usually does what you asked — but it doesn't think through (a) what edge cases the new feature needs to handle, or (b) which other features in your codebase might break as a side effect. So the agent ships code that works for the happy path, breaks an unrelated feature, or misses an obvious edge case. You discover the problem hours later, after the agent has built more code on top of the broken foundation.

Tether prevents this by intervening in three places:

  1. Before the change: when Claude Code is about to make a change, tether predicts which other features will be affected and generates the edge cases the new feature needs to handle. Claude Code reads this analysis and either incorporates it, or asks the user to make decisions about ambiguous cases.
  2. During the change: a continuous file watcher monitors the codebase. If Claude Code edits a file, tether runs the tests for every feature whose code touches that file, plus an LLM-based intent check. Drift gets caught immediately.
  3. After the change: tether updates a feature ledger — a structured YAML file that is the source of truth for what features exist, what they do, what status they're in, and which files they live in. The ledger is the project's memory.

The result is a coding workflow where the agent thinks before it acts, the user gets forced clarity about what they're changing and why, and unintended regressions get caught the moment they happen.

What tether is NOT. It is not an agent. It does not write code itself. It does not replace tests, linters, or CI. It does not block Claude Code from doing anything — it provides analysis, the user decides. Keep this scope sacred. Every feature creep idea, ask: "does this serve the impact-analysis or edge-case-generation loop, or am I trying to turn tether into something else?" If the latter, drop it.

2. The core insight: cheap watcher + expensive worker

This is the architectural premise that makes the whole project economically viable. Understand it before designing anything else.

Coding agents like Claude Code use a large model (Sonnet/Opus) for the actual work — reading code, planning changes, writing functions. That work is creative and needs the big brain. But tether's job is fundamentally different. Tether asks structured questions like "which features touch file X?", "what edge cases apply to a rate limiter?", "does this diff align with the ledger?" These questions don't need creative reasoning — they need structured pattern matching.

A small fast model — Claude Haiku 4.5 — can answer them reliably at roughly 1/3 the cost of Sonnet and 1/15 the cost of Opus. Tether uses Haiku for everything. The expensive worker (Claude Code) does the creative work; the cheap watcher (Haiku) does the structured checking.

Cost numbers to keep in mind. Claude Sonnet 4.6: $3/M input, $15/M output. Claude Haiku 4.5: $1/M input, $5/M output. A typical tether check should cost $0.005–0.02 depending on complexity. A typical Claude Code turn costs $0.05–0.15. Tether's overhead per session should land around 15–25% of the worker's cost. Higher than a simple drift watcher would be, because impact analysis and edge case generation are richer operations — but still cheap enough that the math works.

The hard rule: the watcher model must be Haiku or equivalent, never Sonnet or Opus. If you find yourself reaching for the big model in tether's path, stop and reconsider. You are breaking the economics.

3. The two pillars: impact analysis & edge case generation

Tether does two things that no existing tool does. Both are powered by the feature ledger (Section 4) and the cheap watcher.

Pillar 1: Impact analysis

Before Claude Code makes a change, tether answers the question: "what else in this codebase will be affected by this change, and how badly?"

The mechanism: every feature in the ledger is tagged with the files it lives in. When Claude Code is about to modify file X, tether looks up which features list X in their files array. Those features are the blast radius. Tether then asks Haiku to analyze the planned change against each affected feature and return:

  • Risk level per feature: none, low, medium, high
  • Reasoning: one sentence per feature explaining why
  • Suggested mitigations for medium/high risks

The output gets written to a file Claude Code reads on its next turn. If any risk is medium or high, Claude Code is instructed (via CLAUDE.md) to ask the user before proceeding. This is the proactive intervention — drift is prevented, not caught after the fact.

Example interaction:

User: "Add caching to the rate limiter for performance."

Claude Code: [reads ledger, sees rate limiter in f2 and middleware in f5]
            [calls tether plan src/ratelimiter.py]

Tether → Haiku: "User wants to add caching to ratelimiter.py. Affected
                features: f2 (rate limiting), f5 (request middleware),
                f8 (admin override). What's the risk to each?"

Haiku → Tether: f2: medium — caching may bypass per-user windows
                f5: low — middleware reads ratelimiter output, behavior changes
                f8: high — admin override depends on real-time state, caching breaks it

Claude Code: "Before I add caching, I need to flag two concerns:
              1. Medium risk to rate limiting itself — caching could bypass
                 the per-user sliding window. How fresh do you need the data?
              2. HIGH risk to admin override (f8) — that feature depends on
                 real-time state. Caching will likely break it.

              Should I proceed anyway, skip caching for admin paths, or
              rethink the approach?"

That conversation is the entire value proposition of tether. The agent caught the problem before writing a single line of code.

Pillar 2: Edge case generation

When Claude Code is about to add a new feature, tether generates the edge cases that feature needs to handle, weighted by realistic frequency. Edge cases are stored on the ledger entry for that feature, so future changes to the feature can be re-validated against them.

Frequency labels:

  • high — occurs in >1% of operations
  • medium — 0.01%–1%
  • low — <0.01%, but a known real scenario
  • negligible — theoretically possible but vanishingly rare

Each edge case gets a must_handle: true flag if it's high or medium, OR if it's low and the rationale mentions catastrophic/security/data-loss outcomes. This rule is applied programmatically; Haiku proposes, the rule decides. Do not let the model set must_handle directly — it will be too generous.

For each must_handle: true edge case, tether also generates a pytest test and saves it to .tether/tests/feature_<id>/test_*.py. These tests are the "regression armor" for the feature. Whenever the feature's files are touched, these tests run.

4. The feature ledger (central concept)

The ledger is the most important single concept in tether. It is a YAML file at .tether/ledger.yaml that serves as the project's source of truth for:

  • What features exist
  • What status each feature is in (active, building, removing, deprecated)
  • Which files each feature lives in
  • What edge cases each feature must handle
  • The history of when each feature was added, modified, or removed, and why

The ledger is jointly maintained by the user and Claude Code. Tether enforces that changes to the ledger happen before changes to the code. The ledger is the contract; the code follows.

Schema

# .tether/ledger.yaml
version: 4                                # incremented on every change
last_updated: 2026-04-11T09:30:14Z
features:
  - id: f1
    name: "Per-user rate limiting"
    description: "100 req/min per user, sliding window"
    status: active                        # active|building|removing|deprecated
    files:
      - src/ratelimiter.py
      - src/middleware.py
    edge_cases:
      - id: e1
        description: "User exceeds limit within window"
        frequency: high
        must_handle: true
        test_file: .tether/tests/feature_f1/test_exceeds_limit.py
      - id: e2
        description: "User's first request"
        frequency: high
        must_handle: true
        test_file: .tether/tests/feature_f1/test_first_request.py
      - id: e3
        description: "Clock skew between servers"
        frequency: low
        must_handle: true                 # low + catastrophic = must_handle
        test_file: .tether/tests/feature_f1/test_clock_skew.py
    history:
      - action: added
        at: 2026-03-15T10:00:00Z
        reason: "Initial implementation"

  - id: f8
    name: "Admin rate limit override"
    description: "Admins bypass rate limits via JWT claim"
    status: active
    files:
      - src/ratelimiter.py
      - src/auth.py
    edge_cases: [...]
    history: [...]

  - id: f12
    name: "Redis-backed cache for user lookups"
    status: removing
    files:
      - src/cache.py
    history:
      - action: added
        at: 2026-03-20T14:00:00Z
        reason: "Performance"
      - action: marked_for_removal
        at: 2026-04-11T09:15:00Z
        reason: "Switching to in-memory dict for simplicity"

Why this schema matters

Three properties make this schema work:

Files per feature. This is what enables targeted testing. When file X changes, tether queries: "which features list X in their files array?" Returns a small set (usually 1–3 features). Tether runs tests only for those features. Tests stay fast even as the project grows. Edge case: shared files (like utils.py) listed by 8 features will run all 8 — that's correct behavior, because a shared util change really can break 8 things.

Edge cases per feature. This is what enables ongoing regression protection. The edge cases generated when a feature was first added become the test suite that protects the feature forever. When the feature's code is touched, those tests run. If a test fails, tether knows exactly which feature is broken and which edge case it's failing on.

History per feature. This is what enables drift attribution. When something breaks, tether walks the history to figure out what recent change caused it. The history also serves as the project's memory — the user can ask tether log f8 and get the full evolution of feature f8.

5. The four architectural rules

These rules keep tether from painting itself into a corner. Follow all four, even when they feel like overkill.

Rule 1: Watch the file system, not the agent

Tether must never integrate with Claude Code's internals or hooks. It watches the project directory using watchdog. This is the universal interface — every coding agent edits files, so tether automatically works with any agent (Codex, Gemini CLI, Aider, Cursor) later, with zero per-agent integration code.

Resist the temptation to use Claude Code's hooks system. Hooks would let you block bad edits before they happen, but they tie tether to Claude Code specifically. v0.1 sticks to "watch and analyze and report" — strictly observational interventions delivered through the notes file mechanism.

Rule 2: All file paths and integration points are configurable

Anywhere tether writes to CLAUDE.md, it instead writes to config.notes_file. Anywhere it reads tests from .tether/tests/, it instead reads from config.tests_dir. v0.1 has hardcoded defaults; the config indirection just exists for v0.2 onward when other agents need different defaults.

Rule 3: The watcher model is behind a small interface

Do not call the Anthropic SDK directly from tether's logic. Define a tiny interface — WatcherModel with one method, check(system, user, schema) -> dict — and put one implementation behind it (the Haiku-backed one). When someone later wants to swap in Gemini Flash or local Ollama, it's a 30-line addition, not a refactor.

# tether/watcher_models/base.py
from typing import Protocol

class WatcherModel(Protocol):
    def check(self, system: str, user: str, schema: dict | None = None) -> dict:
        """Run a single watcher check. Return a structured verdict."""
        ...

Rule 4: No Claude-Code-specific features in v0.1

No hook integration, no parsing of Claude Code's internal state, no scraping of its terminal output. Every feature must work via the universal "read ledger / watch files / write notes" pattern. This is what keeps tether buildable as universal infrastructure later.

6. v0.1 scope

v0.1 is intentionally ambitious. Both pillars (impact analysis and edge case generation), feature ledger, four phases (bootstrap, plan, watch, verify), Claude Code integration via CLAUDE.md. This is a real project, not a weekend sketch.

In scope:

  • Feature ledger with full schema and versioning
  • CLI: tether init, tether bootstrap, tether plan, tether watch, tether verify, tether ledger, tether log, tether status, tether report
  • Bootstrap phase: scans existing codebase, proposes initial ledger, user reviews
  • Plan phase: takes a planned change, returns impact analysis (which features at what risk) and (for new features) edge case enumeration
  • Watch phase: file watcher, targeted test runs by affected features, intent check, drift notes
  • Verify phase: post-change, re-runs all affected feature tests, updates ledger, writes session summary
  • Auto-generated CLAUDE.md instructions that teach the agent how to interact with tether
  • Anthropic Haiku as the default watcher model
  • Reproducibility: every session writes a JSONL log of every model call with cost tracking
  • Default: Python projects (tree-sitter Python parser)

Out of scope (v0.2+):

  • Other agents (Codex, Gemini CLI, Aider) — architecture supports it, no implementation
  • Other watcher models — interface supports it, no implementation
  • JS/TS/Go/Rust — Python only in v0.1
  • Hook-based blocking interventions
  • Web dashboard / HTML reports
  • Multi-repo / monorepo features beyond a single project root
Scope discipline. This is already a large project. Resist adding anything not on the in-scope list. If you find yourself thinking "I should also add X for completeness," write it in TODO.md and keep going. The biggest risk is overscoping, not technical difficulty.

10. Phase 1 — Bootstrap (one-time)

Bootstrap runs once per project via tether bootstrap. It builds the initial feature ledger by scanning the existing codebase. This is the only phase that operates without an existing ledger.

Step 1.1 — Static scan

Walk all .py files in the project root. For each file, use tree-sitter to extract:

  • Top-level functions and their signatures
  • Top-level classes and their public methods
  • Module-level docstring (if present)
  • File path → symbol map

This is pure static analysis, no LLM calls. Output: an in-memory representation of the codebase structure.

Step 1.2 — Feature proposal (LLM)

Send the static scan to Haiku in chunks (one chunk per directory or per ~3000 lines of code) with the BOOTSTRAP_FEATURE_PROPOSAL prompt. Haiku proposes feature definitions: name, description, files, and a guess at edge cases.

Aggregate the proposals across chunks. De-duplicate by name similarity. The result is a draft ledger.

Step 1.3 — User review (mandatory)

Open the draft ledger in $EDITOR. Print clear instructions: "These are tether's guesses about your project's features. Edit freely. Delete features that don't make sense. Merge duplicates. Add features tether missed. The accuracy of everything tether does later depends on this ledger being correct."

After save, validate the YAML and write the final .tether/ledger.yaml. Create version 1 in .tether/ledger.history/v1.yaml.

Step 1.4 — Edge case backfill

For each feature in the reviewed ledger, run edge case generation (Phase 2's edge_cases logic) to populate the edge_cases array. Generate the corresponding pytest test files in .tether/tests/feature_<id>/.

This is the slowest part of bootstrap — one Haiku call per feature for edge cases, plus another for each test. For a 20-feature project, expect ~80 Haiku calls and roughly $0.50–1.00 in cost. Show a progress bar.

11. Phase 2 — Plan (per change)

Plan runs every time Claude Code is about to make a significant change. The agent calls tether plan <file_path> --intent "<description of planned change>" and tether returns a structured analysis written to .tether/PLAN.md, which Claude Code then reads on its next turn.

Step 2.1 — Identify affected features

Query the ledger: which features list file_path in their files array? Return them as the affected feature set.

Step 2.2 — Impact analysis

For the affected features, send a single Haiku call with the IMPACT_ANALYSIS prompt: the planned change description, the file path, the affected features (with their descriptions and edge cases), and the recent diff if available. Haiku returns:

{
  "risks": [
    {"feature_id": "f2", "level": "medium", "reason": "...", "mitigation": "..."},
    {"feature_id": "f5", "level": "low", "reason": "...", "mitigation": null},
    {"feature_id": "f8", "level": "high", "reason": "...", "mitigation": "..."}
  ],
  "overall_risk": "high"
}

ask_user_required is computed by tether (not Haiku) using the rule: any risk level >= config.plan.ask_threshold.

Step 2.3 — Edge case generation (only for new features)

If the intent describes adding a new feature (not modifying an existing one), also run edge case generation. Same logic as bootstrap step 1.4. Output gets included in the plan notes.

Step 2.4 — Write plan notes

Write the structured analysis to .tether/PLAN.md in markdown. If ask_user_required is true, include a clear "STOP — ask the user about these risks before proceeding" header. Otherwise, format it as advisory information for Claude Code to consider.

Example PLAN.md output:

# Tether Plan — 2026-04-11 09:32:14

**Planned change:** Add caching layer to ratelimiter.py for performance
**Affected file:** src/ratelimiter.py

## ⚠️ STOP — Ask user about these risks before proceeding

### High risk: f8 (Admin rate limit override)
The admin override depends on real-time rate limit state. Adding a
cache layer will likely break it because cached values won't reflect
real-time admin grants.

**Mitigation:** Skip caching for admin code paths, or invalidate cache
on admin override events.

### Medium risk: f2 (Per-user rate limiting)
Caching may bypass the per-user sliding window logic if cache TTL
exceeds the window size.

**Mitigation:** Set cache TTL strictly less than the rate limit window.

### Low risk: f5 (Request middleware)
Middleware reads rate limiter output. Behavior may shift if cached
values are stale.

## Decision required

Before proceeding, please ask the user:
1. Should caching apply to admin paths, or skip them?
2. What freshness requirement do you have for rate limit data?

12. Phase 3 — Watch (continuous)

Watch is the long-running process started by tether watch. It uses watchdog to monitor the project, runs targeted checks on every file change, and writes drift notes when something breaks.

Step 3.1 — File event handling

Standard watchdog setup. Filter through ignore_globs, file extension whitelist (.py, .md, .yaml, .toml, .json), and per-path debouncer (debounce_ms).

Step 3.2 — Targeted test run

When a file change fires, query the ledger: which features touch this file? For each affected feature, run the tests in .tether/tests/feature_<id>/. Compare results against the previous run for that feature. Flag any newly-failing tests.

This is what makes tether fast even on large projects. You're not running the whole test suite — you're running ~5–20 tests for ~1–3 features. Expect under 10 seconds total.

Step 3.3 — Intent check

Compute a unified diff of the changed file. Send to Haiku with the affected features (from the ledger) and ask: "does this diff align with the current state of these features?" Possible verdicts: aligned, neutral, drifted, looks_intentional.

looks_intentional is for the deliberate-removal case: the diff broke a test, but it looks like a clean intentional change (the user is removing a feature). Tether responds with a soft note suggesting the user update the ledger, instead of a hard "you broke this" alarm.

Step 3.4 — Static invariant check

Re-parse the changed file with tree-sitter. Compare against the snapshot from the bootstrap. Flag if a function signature changed or an exported symbol disappeared. This is free and instant.

Step 3.5 — Verdict aggregation & whisper

Combine the three checks. Severity:

  • None — all checks pass. Silent.
  • Soft — intent says drifted, tests still pass. Gentle note.
  • Intentional-looking — tests broke, intent says it looks intentional. Polite "did you mean to do this? if so update the ledger."
  • Hard — tests broke, intent didn't say intentional. Firm note: "you broke test X, which protects feature Y."
  • Critical — multiple feature tests broke. Firm note + drift attribution.

Notes are written to .tether/DRIFT.md, with the worker's main notes file (CLAUDE.md) containing a one-line pointer ("see .tether/DRIFT.md for current drift status"). This keeps CLAUDE.md clean and stable while still reaching the agent on its next turn.

Step 3.6 — Diff chunking for large changes

If a diff exceeds ~300 lines, split it by function/class boundary using tree-sitter and run the intent check on each chunk separately. Each chunk is small enough for Haiku to reason about clearly. Aggregate verdicts — if any chunk drifts, the whole change is flagged with the specific chunk called out. Cost for an 800-line change with chunking: roughly $0.012 (about 4 chunks at $0.003 each).

13. Phase 4 — Verify (after change)

Verify runs after Claude Code finishes a logical unit of work. The agent calls tether verify, which:

  1. Re-runs all tests for every feature whose files were modified in this session
  2. Re-validates the ledger schema
  3. Updates the ledger with any status changes implied by the session (e.g., features marked building get re-checked for completion)
  4. Writes a session summary to .tether/reports/<run_id>.md
  5. Snapshots the ledger to .tether/ledger.history/v<N+1>.yaml

The session summary includes: features modified, tests run and their results, drift events with attributions, total cost spent, ledger version delta. This is what the user reads to understand what happened in the session.

14. The Claude Code integration (CLAUDE.md instructions)

This is what makes tether actually work in practice. tether init writes a managed block into CLAUDE.md that teaches the agent how to interact with tether. The exact text:

<!-- TETHER:INSTRUCTIONS START — managed by tether, do not edit by hand -->
## Tether is active in this project

A tool called **tether** is helping coordinate changes in this project.
Tether maintains a "feature ledger" at `.tether/ledger.yaml` that tracks
every feature, its files, its edge cases, and its status. Before making
changes, you should consult tether and follow its guidance.

### Workflow rules

**Before making any non-trivial code change:**

1. Read `.tether/ledger.yaml` to see the current feature set.

2. Run `tether plan <file_path> --intent "<short description>"` for
   the file you're about to modify. This generates impact analysis at
   `.tether/PLAN.md`.

3. Read `.tether/PLAN.md`. If it contains "STOP — Ask user", **stop and
   ask the user about the risks before proceeding**. Do not make the
   change until the user has answered.

4. If the change adds a new feature, the plan output will also contain
   suggested edge cases. Implement the code to handle the ones marked
   `must_handle: true`.

**Before removing or replacing a feature:**

1. Edit `.tether/ledger.yaml` to mark the feature `removing`.
2. Then change the code.
3. After the change, run `tether verify` to update the ledger.

**Always check `.tether/DRIFT.md`** at the start of each turn. If it
contains a drift note, the most recent change broke something. Read the
note carefully and either fix the issue or, if the change was intentional,
update the ledger to reflect the new reality.

**After finishing a logical unit of work, run `tether verify`.** This
re-validates everything and updates the ledger.

### Why this matters

Tether catches two classes of mistakes that are otherwise easy to miss:

1. **Cross-feature regressions** — when a change to feature A
   accidentally breaks feature B because they share code.
2. **Missed edge cases** — when a new feature handles the happy path
   but misses common edge cases.

The cost of consulting tether (a few seconds per change) is much less
than the cost of discovering these mistakes hours later.

16. Recommended build order

Build in this order. Each step should leave the project in a runnable state. Do not skip ahead.

  1. Project skeleton. pyproject.toml, package layout, cli.py with all commands as stubs printing "not implemented." pip install -e . works, tether --help prints.
  2. Config. Pydantic models. tether init writes a working config and the CLAUDE.md instructions block. Round-trip test.
  3. Ledger schema & store. Pydantic models for the ledger. Read/write/version. Validation. Pure functions, no LLM.
  4. Ledger queries. "Which features touch file X?" "What's the test directory for feature Y?" "What edge cases does feature Z have?" Pure functions over the ledger.
  5. State directory. .tether/ creation, manifest writer, JSONL log writer with cost tracking. Session ID generation.
  6. Watcher model interface. WatcherModel protocol and the Haiku implementation. Tool-use response handling. Test against a sample prompt.
  7. Bootstrap — static scanner. Tree-sitter setup, walk project, extract symbols, build the file → symbol map. Pure code, no LLM.
  8. Bootstrap — feature proposal. Chunk the static scan, send to Haiku with the BOOTSTRAP prompt, aggregate, deduplicate. Write draft ledger.
  9. Bootstrap — review flow. Open ledger in $EDITOR, validate on save, write final ledger. Print clear instructions.
  10. Edge case generation (shared module). Given a feature, generate edge cases via Haiku with the EDGE_CASE prompt. Apply must_handle rule.
  11. Test generation (shared module). Given an edge case, generate a pytest test via Haiku with the TEST_GEN prompt. Write to .tether/tests/feature_<id>/. Run --collect-only sanity check.
  12. Bootstrap — edge case backfill. Loop over features, run edge case generation and test generation for each. Progress bar. tether bootstrap end-to-end works.
  13. Plan — impact analysis. Query ledger for affected features, send to Haiku with IMPACT_ANALYSIS prompt, parse risks, compute ask_user_required.
  14. Plan — notes writer. Format the impact analysis (and edge case enumeration if applicable) as markdown. Write to .tether/PLAN.md.
  15. Plan — CLI command. tether plan <file> --intent "..." end-to-end. Test against a sample project.
  16. Watch — file events & debounce. Watchdog setup, glob filters, debouncer. Print events to console.
  17. Watch — ledger reload. Separate watcher on .tether/ledger.yaml. When it changes, snapshot and reload in-memory.

  19. Watch — intent check. Compute diff, send to Haiku with INTENT_CHECK prompt, parse verdict (including looks_intentional). Implement diff chunking for diffs >300 lines.
  20. Watch — invariant check. Re-parse changed file with tree-sitter, compare against snapshot.
  21. Watch — verdict aggregation. Combine three checks, compute severity, log to JSONL.
  22. Watch — drift notes writer. Markdown formatter for each severity level. Marker-based upsert into .tether/DRIFT.md.
  23. Watch end-to-end. tether watch runs the full pipeline. Manual test against a sample project.
  24. Verify — runner. Re-run all affected feature tests, validate ledger, snapshot, write summary. tether verify works.
  25. Drift attribution. History walker, attribution prompt, fix-impact prompt. Append to drift notes.
  26. Ledger CLI commands. tether ledger (rich table view), tether log [id] (history view).

  28. Cost tracking & budget cap. Sum costs from JSONL, halt watch loop on budget exceeded, surface in status.

18. Competitive landscape

Important context: tether is in a clear gap. The Claude Code guardrail ecosystem as of now is almost entirely focused on security, not on intent or impact. Knowing this is useful both for positioning and for not accidentally re-building something that already exists.

What exists:

  • Built-in Claude Code permissions — bash command approval, sandbox mode, deny-list paths. Reactive, focused on "don't let it delete my home directory."
  • Hooks-based security tools — Rulebricks, Codacy Guardrails, dwarvesf/claude-guardrails, NextWork. All variations on "block dangerous bash commands" and "scan files for vulnerabilities or secrets after edits."
  • Linters and formatters — run via PostToolUse hooks. Catch style issues and obvious bugs.

What does not exist:

  • Cross-feature impact analysis — predicting which other features get hit by a change before the change is made.
  • Proactive edge case generation — telling the agent what edge cases the feature it's about to add needs to handle.
  • A persistent feature ledger — a structured, agent-readable source of truth for what features exist in a project.
  • Intent-level drift detection — catching "the agent stopped doing what the user asked for" as opposed to "the agent ran a dangerous command."

Tether occupies the second category entirely. Position the README accordingly: "Codacy catches insecure code. Linters catch broken code. Tether catches off-task code and unintended regressions."

One blog post (sshh.io, "How I Use Every Claude Code Feature") makes the case explicitly that block-at-write hooks are bad UX because they confuse the agent mid-plan. Tether's "watch and write notes for the next turn" approach aligns with this philosophy.

19. Known limitations

Be honest about these in the README.

  • Edge case frequency estimation is approximate. Haiku will sometimes mis-rank. The user-review step in bootstrap is the mitigation.
  • Impact analysis is heuristic, not provable. Haiku is doing pattern-matching, not formal program analysis. It will occasionally miss real impacts and occasionally flag false positives. The "ask the user" step is the safety net.
  • Bootstrap quality depends on user review. If the user rubber-stamps the bootstrap-proposed ledger without editing, the ledger will be wrong, and everything downstream will be less useful. The CLI must make this clear.
  • No protection against catastrophic destructive operations. Tether watches and warns; it does not block. If the agent runs rm -rf, tether finds out after the fact. Users who need destructive-operation protection should pair tether with security guardrails like Rulebricks or Claude Code's built-in permissions.
  • Python projects only. Tree-sitter and pytest assumptions are Python-specific. Other languages are v0.2+.
  • Single-repo only. Monorepo support is v0.2+.
  • Non-determinism. Two runs of the watcher on the same change may return slightly different verdicts. Inherent to LLM checking.
  • Cost is higher than simple drift detection. Expect 15–25% overhead vs. unguarded Claude Code sessions. The richness of impact analysis costs more than passive watching. Worth it, but be transparent.
